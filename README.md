# mlops-cicd-sample

End-to-end **CI/CD for ML** demonstration using **GitHub Actions** + **GitHub Container Registry (ghcr.io)**. Trains a tiny fraud-detection model, packages it as a FastAPI service in Docker, and pushes versioned images on every merge to `main`. PRs are gated by a model **accuracy threshold** — drop below 90% accuracy and the merge is blocked.

This is a **standalone exercise that lives in its own dedicated GitHub repo**. The model itself is intentionally minimal — it's a sklearn pipeline, just so the CI/CD pipeline has something real to operate on.

---

## Why CI/CD for ML is different from CI/CD for normal apps

Classical CI/CD checks compile + unit-test + deploy. ML adds three things:

| ML-specific concern | What the pipeline does about it |
|---|---|
| **Training is non-deterministic by default** | Pin `random_state=42` in `train.py`; the test suite asserts reproducibility |
| **Model quality regression** | Accuracy threshold in `tests/test_train.py` blocks merges if a code change degrades the model |
| **Models are big artifacts** | The trained `model.joblib` is **regenerated by CI** every release (NOT committed to Git) — same code → same model |
| **Data + code coupling** | A change to the training-data generator changes the model. CI re-trains, doesn't trust a stale checked-in artifact. |

The "accuracy gate" is the heart of this exercise. It's the simplest form of model CI — a PR that breaks the model fails the build, period.

## Repository layout

```
mlops-cicd-sample/
├── .github/
│   └── workflows/
│       ├── ci.yml              ← PR checks: lint + tests + accuracy gate
│       └── release.yml         ← On merge: train + test + build + push image
├── src/
│   ├── __init__.py
│   ├── train.py                ← Train + save model.joblib
│   └── server.py               ← FastAPI server
├── tests/
│   ├── test_train.py           ← Including the accuracy gate
│   └── test_server.py          ← FastAPI smoke tests
├── Dockerfile                  ← Multi-stage, non-root, bakes model.joblib
├── requirements.txt            ← Runtime deps
├── requirements-dev.txt        ← + pytest, ruff, httpx
├── pyproject.toml              ← Ruff + pytest config
├── .gitignore                  ← excludes model.joblib (CI regenerates it)
└── README.md
```

`model.joblib` is **never committed** — CI rebuilds it from `train.py` on every release. This matches production practice: model artifacts live in a registry (MLflow, S3), not Git.

## The two workflows

### `ci.yml` — runs on every PR + push to main

```
checkout → setup Python 3.11 → install → ruff → pytest
```

Catches:
- Syntax errors
- Import failures
- Style violations (`ruff check`)
- Unit test failures
- **Accuracy regression** — `test_accuracy_above_threshold` fails the build if accuracy < 90%

Free runner-minutes for public repos. ~2-3 minutes per run.

### `release.yml` — runs ONLY on push to main

```
checkout → train model → run tests → docker build → push to ghcr.io
```

After a PR merges:
1. Re-runs training and tests as defense-in-depth (the accuracy gate isn't bypassable)
2. Builds the Docker image (multi-stage, ~250 MB final)
3. Pushes to `ghcr.io/<owner>/mlops-cicd-sample` with two tags:
   - `sha-abc1234` (commit SHA, immutable)
   - `latest` (mutable, convenience)

GitHub's built-in `GITHUB_TOKEN` has package write permission when `permissions.packages: write` is set in the workflow — **no extra secrets needed** for ghcr.io.

## Set up the GitHub repo (one-time)

```powershell
cd exercises/mlops-cicd-sample
git init -b main
git add .
git commit -m "initial commit"

# Create the public repo on GitHub
gh repo create asafwat/mlops-cicd-sample --public --description "CI/CD for ML pipeline demo (GitHub Actions + ghcr.io)"
git remote add origin https://github.com/asafwat/mlops-cicd-sample.git
git push -u origin main
```

Once pushed:
1. GitHub Actions starts the `CI` workflow immediately — green check should appear within 3-4 minutes
2. The `Release` workflow also runs (because the initial commit is to `main`) — produces an image at `ghcr.io/asafwat/mlops-cicd-sample:latest`
3. Make the package public via GitHub UI: repo → Packages → mlops-cicd-sample → Settings → Change visibility → Public. Otherwise `docker pull` requires auth.

## Local development loop

```powershell
# Train a model + run the full test suite locally
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements-dev.txt

python -m src.train
pytest tests/ -v
ruff check src/ tests/

# Run the FastAPI server locally
uvicorn src.server:app --host 0.0.0.0 --port 8080

# Build the image
docker build -t mlops-cicd-sample:0.1 .
```

To deploy locally on Kind (consistent with the rest of the lab):
```powershell
kind load docker-image mlops-cicd-sample:0.1 --name mlops
# Then write a Deployment + Service + Ingress (pattern from any of the other lab exercises)
```

To **pull from ghcr.io** instead (the CI-built image):
```powershell
docker pull ghcr.io/asafwat/mlops-cicd-sample:latest
docker tag  ghcr.io/asafwat/mlops-cicd-sample:latest mlops-cicd-sample:latest
kind load docker-image mlops-cicd-sample:latest --name mlops
```

## Demoing the accuracy gate

Open a PR that intentionally tanks the model — e.g., change `n_estimators=100` to `n_estimators=1` in `src/train.py`. Push:

```powershell
git checkout -b break-the-model
sed -i 's/n_estimators=100/n_estimators=1/' src/train.py
git commit -am "intentional regression"
git push -u origin break-the-model
gh pr create --fill
```

GitHub Actions runs the CI workflow → `test_accuracy_above_threshold` fails → PR shows a red ❌ → merge blocked. This is the **model promotion gate** working as designed.

Revert the change, push again, watch the check go green. Now the merge is unblocked.

## What this exercise demonstrates

| Skill | How |
|---|---|
| GitHub Actions workflows | Two workflow files (`ci.yml` + `release.yml`) covering both PR-time + release-time |
| Test-driven ML | `tests/test_train.py` asserts model invariants, not just code correctness |
| Model promotion gates | Accuracy threshold blocks degraded models from being released |
| Container registry integration | ghcr.io via built-in `GITHUB_TOKEN`, no secret management |
| Docker layer caching | `cache-from: type=gha` cuts repeated build time by ~70% |
| Reproducible builds | `random_state=42` + `requirements.txt` pinning = same code → same model |
| Multi-stage Dockerfile + non-root user | Production-shape image |

## Production extensions (where to go from here)

| Extension | What it adds |
|---|---|
| **Trigger an Airflow DAG from CI** | On image push, call Airflow REST API to start a retraining/eval DAG that consumes the new image. Requires Airflow REST API auth + cluster-reachable endpoint. |
| **Kind pulls from ghcr.io** | Replace `kind load` with native pulls. Configure Kind's containerd with ghcr.io credentials (via `--config` kind-config.yaml with `containerdConfigPatches`). |
| **Model registry promotion** | Instead of (or alongside) ghcr.io, log the model to MLflow Model Registry; the accuracy gate promotes the version to the "Staging" stage. KServe pulls from the registry instead of the image. |
| **GitOps deploy** | On image push, CI updates an image tag in a separate "deployment manifests" repo. ArgoCD detects the change and syncs the new image to the cluster. |
| **Champion-vs-challenger comparison** | Instead of an absolute accuracy threshold, compare against the production model's accuracy on a fixed eval set. Block regressions, not absolute drops. |
| **Data validation** | Add a step that validates the training data distribution before training (Great Expectations, custom checks). |
| **Image vulnerability scanning** | `trivy` or `grype` as a release-workflow step; fail on critical CVEs. |
